Autoscaling terms

Term Definition
Autoscaling/Auto Scaling Automatic changes that add or remove compute resources when certain criteria is met and replacing resources that are not healthy
Horizontal Scaling Adding more nodes or workers a.k.a. scaling in and scaling out
EC2 Spot Fleet A mixed compute model that manages Spot and On-Demand EC2 instances
Vertical Scaling Adding more resources to the system (memory and CPU) a.k.a. scaling up and down

Autoscaling recap

Key Points

  • Designed to to maintain stable performance and control costs

  • Scaling can add/remove system resources (vertical scaling) or add/remove nodes or workers (horizontal scaling)

  • Amazon EC2 Autoscaling uses health notifications and instance status checks to determine the health status of an instance. If an instance is marked unhealthy, it is scheduled for replacement.

  • Scaling for all AWS resources is done in the Auto Scaling interface

  • Scalable AWS resources include

    • EC2
    • EC2 Spot Fleets
    • ECS
    • DynamoDB
    • AWS Aurora

Think Like a Cloud Architect

QUESTION:

Reflect

What is the difference between Scaling Out and Scaling up? Do you believe they accomplish the same objective?

ANSWER:

Scaling out is the process of adding more resources to spread out a workload. An example would be adding servers to a load balancer to share a workload. Scaling up is the process of making a resource larger and more powerful in response to an increased workload. An example would be adding RAM or CPUs to a server so it can manage an increase in demand.

Scaling out is also referred to as horizontal scaling

Scaling up is also referred to as vertical scaling

How To Set up an Auto Scaling Group in AWS

  1. Login to the AWS console at console.amazon.aws.com
  2. Browse to the EC2 Management page
  3. Locate and select Auto Scaling Groups in the left menu
  4. Run the wizard to create an Auto Scaling Group (Launch Configuration + Auto Scaling Group)
  5. Create and name a Launch Configuration using the free tier instances and default storage.
  6. Save the Launch Configuration
  7. Create and name your Auto Scaling Group
  8. Save your Auto Scaling Group and wait for it to launch.
  9. Return to the Auto Scaling menu and view your launch configuration and Auto Scaling Group.

If you have set up your Auto Scaling Group for practice, don't forget to delete your launch configuration and Auto Scaling Group and logout of the AWS Console when you are done.

What do you think? What would be the desired outcome of running your single EC2 instances in this type of auto scaling group? Would you consider this configuration to be highly available or fault-tolerant?

ANSWER:

Depending on how your company approaches costs vs. availability, you can use an autoscaling group of 1 to make sure that if your EC2 instance experiences problems and can’t perform properly, it will be replaced by the autoscaling group, and your application will experience minimal downtime.
